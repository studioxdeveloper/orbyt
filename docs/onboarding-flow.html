Steg med tankegang

1 Registrering / Logg inn

Bruker oppretter konto med e-post eller logger inn. Autentisering håndteres av Orbyt (Convex/Supabase Auth). Facebook/Google som login-provider er valgfritt for MVP.

Tankegang

Vi starter med e-post + passord for å minimere avhengigheter til Meta/Google i fasen før OAuth er klar. Bruker oppretter én Orbyt-konto som senere kobles til flere plattformer. Enkel registrering = lavere terskel for å prøve appen.

Referanse: app.html (login-skjerm i prototype) · API: POST /api/auth/register, POST /api/auth/login

2 Velkommen / Introduksjon

Valgfritt velkomstskjerm som forklarer verdien av Orbyt: «Publiser til alle plattformer fra ett sted.» Kan inneholde kort intro til Orbit-visualisering og neste steg (koble plattformer).

Tankegang

Vi vurderer en kort velkomst for nye brukere, men ønsker ikke lang «tutorial» som blokerer. Alternativ: Gå rett til Feed med CTA «Koble plattformer» hvis ingen kontoer er tilkoblet. MVP kan starte uten dedikert velkomst og heller bruke empty state med tydelig CTA.

Referanse: Kan være ny skjerm eller utvidet empty state på Feed

3 Koble plattformer (OAuth)

Bruker klikker «Koble til» for Instagram, Facebook eller YouTube. Omdirigeres til Meta/Google OAuth. Etter godkjenning returneres bruker til Orbyt med tilkoblet konto. For Facebook: bruker velger Page(s). For Instagram: krever Professional-konto koblet til Facebook Page.

Tankegang

Plattformer kobles én etter én. Vi viser tydelig status (tilkoblet / ikke tilkoblet) og hvilke kontoer som kan publiseres til. Token-utløp: Meta sine tokens utløper etter 60 dager uten refresh-mulighet. Bruker må koble på nytt. Vi bør varsle før utløp (f.eks. 7–14 dager) og vise enkel «Koble på nytt»-flyt.

Referanse: app.html Profil → «Koble til plattformer» · dashboard.html Tilkoblinger · API: GET /api/auth/oauth/:platform, GET /api/auth/oauth/:platform/callback

4 Opprett innhold

Bruker går til Create-skjermen, velger bilde eller video (opplasting går via Mux for video), skriver caption. Velger hvilke plattformer innlegget skal publiseres til. Kan bruke AI for å generere/forbedre tekst.

Tankegang

Vi viser kun plattformer som er tilkoblet. Bilde lastes direkte; video går via Mux (Direct Upload → transkodering → MP4 URL brukes ved publisering). MVP fokuserer på bilde + caption og video + caption. Avanserte features (musikk, tagging, geolokasjon) vurderes til fase 2. Validering på klientsiden før publisering reduserer feil.

Referanse: app.html Create-skjerm · API: POST /api/videos/upload-url, POST /api/posts

5 Publiser eller planlegg

Bruker velger «Publiser nå» eller «Planlegg innlegg». Ved planlegging velges dato og tid. Backend (cron/scheduler) kjører planlagte poster til valgte plattformer. Status vises per plattform (pending / published / failed).

Tankegang

Planlagt publisering er i MVP-scope — det er en sentral verdi for brukere. Vi bruker server-side scheduling (cron-jobs, kø) siden Meta/YouTube ikke tilbyr native scheduling. Bruker bør se tydelig når posten er planlagt og få varsel ved feil (f.eks. token utløpt).

Referanse: app.html Create «Planlegg innlegg», Kalender · API: POST /api/posts/:id/publish, POST /api/posts/:id/schedule

Integrasjonsrelevante API-kall (onboarding)

  • POST /api/auth/register · POST /api/auth/login
  • GET /api/auth/oauth/:platform → OAuth redirect
  • GET /api/auth/oauth/:platform/callback → Token exchange
  • GET /api/accounts → Liste tilkoblede kontoer
  • POST /api/videos/upload-url → Mux Direct Upload
  • POST /api/posts · POST /api/posts/:id/publish · POST /api/posts/:id/schedule
